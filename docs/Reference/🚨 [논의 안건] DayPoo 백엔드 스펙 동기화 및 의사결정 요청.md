현재 백엔드 상세 설계서(v1.2)와 전체 구현 계획서(v4.3) 간에 일부 스펙 충돌이 확인되었습니다. 원활한 개발 진행과 API 명세(openapi.yaml) 확정을 위해 아래 5가지 핵심 안건에 대한 팀 내 합의가 필요합니다.

## 안건 1. 📍 위치 인증 (GPS) 반경 및 쿨타임 정책

두 문서에서 화장실 방문을 인정하는 GPS 반경과 쿨타임 기준이 다르게 정의되어 있습니다.

- **현재 상황:**
    
    - **설계서(v1.2):** 반경 50m 고정 / 3시간 내 중복 인증 차단
        
    - **기획서(v4.3):** 반경 50~150m 동적 적용 / 최소 체류 시간 1분
        
- **논의 포인트:** 건물 내부나 지하철 등 GPS 음영 지역(Shadow)에서의 오차를 어떻게 처리할 것인가?
    
- **선택지:**
    
    - **A안 (유연성 우선 - 권장):** 동적 반경(50~150m) + 최소 체류 시간(1분) 적용. (GPS가 튀는 현상 방어 가능)
        
    - **B안 (엄격함 우선):** 반경 50m 고정 유지. (어뷰징 방어에 유리하나, 실제 사용자가 인증에 실패할 확률이 높음)
        
    - **추가 결정:** 동일 화장실 '3시간 쿨타임' 제한을 유지할 것인지 여부.
        

## 안건 2. 🔐 인증 시스템 인프라 범위 (자체 가입 vs 소셜 전용)

회원가입 방식에 따라 백엔드 인프라 복잡도가 크게 달라집니다.

- **현재 상황:**
    
    - **설계서(v1.2):** 카카오/구글 OAuth2 소셜 로그인만 명시.
        
    - **기획서(v4.3):** 소셜 로그인 + 자체 회원가입(아이디/비번) + 이메일/SMS 기반 계정 찾기 포함.
        
- **논의 포인트:** 초기 런칭 시 이메일/SMS 발송 인프라(비용 및 연동 공수)를 가져갈 것인가?
    
- **선택지:**
    
    - **A안 (빠른 런칭 - 권장):** 100% 소셜 로그인(카카오/구글)으로 통일. (이메일/SMS 발송 인프라 및 자체 비밀번호 암호화 로직 제거)
        
    - **B안 (풀 스펙 구현):** 기획서대로 자체 회원가입 및 이메일/SMS 연동 포함. (개발 기간 및 AWS SES 등 인프라 설정 공수 추가)
        

## 안건 3. 🤖 AI 이미지 서버 통신 규격

Spring Boot API와 FastAPI(AI 서비스) 간의 배변 이미지 전송 방식이 다릅니다.

- **현재 상황:**
    
    - **설계서(v1.2):** Base64 인코딩 스트링으로 전송.
        
    - **기획서(v4.3):** Byte Array(In-Memory) 바이너리 스트림으로 전송 (No-Save 정책).
        
- **논의 포인트:** 네트워크 페이로드 크기와 메모리 효율성.
    
- **선택지:**
    
    - **A안 (효율성 우선 - 권장):** `multipart/form-data` 형태의 Byte Array로 통일. (Base64 대비 데이터 크기 약 33% 절감, 메모리 해제 용이)
        
    - **B안 (구현 편의성):** Base64 텍스트 기반 전송 유지. (JSON에 바로 태워서 보내기 편함)
        

## 안건 4. ⚙️ 관리자(Admin) API 버전 및 라우팅 규칙

관리자용 API의 엔드포인트 규칙이 일관되지 않습니다.

- **현재 상황:**
    
    - 기획서엔 `/api/v2/admin/dashboard` (v2 사용), 설계서엔 `/api/v1/toilets/sync` (v1 사용)로 기재됨.
        
- **논의 포인트:** 관리자 API의 버저닝과 URL 네임스페이스 격리.
    
- **결정 제안:** 모든 관리자 권한 API는 프론트엔드 라우팅과 맞추어 **`/api/v1/admin/...`** 하위로 일괄 통일 (예: `/api/v1/admin/toilets/sync`).
    

## 안건 5. 🧹 테이블 명칭 및 로컬 DB 툴 통일

문서 내부의 용어 혼용 및 환경 설정 가이드 오류를 바로잡아야 합니다.

- **현재 상황:**
    
    - 방문 기록 테이블 이름이 `VISIT_LOGS`와 `POO_RECORDS`로 혼용됨.
        
    - 로컬 DB 툴로 `pgAdmin4`를 제거(ADR)한다고 했으나, 로드맵(Phase 1)에는 설치한다고 기재됨.
        
- **결정 제안:**
    
    - 인증 기록 테이블 명칭은 **`POO_RECORDS`**로 완전 통일.
        
    - 로컬 환경은 컨테이너 경량화를 위해 **pgAdmin4 설정 항목을 제거하고 DBeaver 사용**으로 가이드 통일.
